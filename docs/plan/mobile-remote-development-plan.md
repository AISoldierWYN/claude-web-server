# 手机端远程开发控制计划

## 实施状态

第一版已落地：

- 默认关闭的功能开关：`[features] mobile_remote_development = false`
- 白名单配置：`claude_web_projects.config.json`
- 示例配置：`claude_web_projects.config.example.json`
- 后端 API：`/api/dev/projects`、绑定/解绑项目、状态、diff、运行预设测试、停止任务
- 开发模式下 CLI `cwd` 指向真实项目目录，会话 cache 仍保存记忆和运行状态
- 前端入口：顶部“开发项目”按钮、项目选择面板、项目状态栏、Diff/运行测试/停止/退出
- 冒烟测试已覆盖默认关闭、启用后白名单读取、绑定项目、状态、diff 和预设测试

## 背景

目标是在当前 Web 版 Claude Chat 中增加一个“开发项目”能力：用户可以在手机浏览器里选择 PC 上白名单内的代码项目，让 AI 在 PC 本地执行阅读、修改、测试、提交等开发工作。手机端只负责展示和控制，真实代码仍保存在 PC 上，真实执行也发生在 PC 上。

这不是普通聊天功能的简单扩展。普通聊天强调会话目录隔离和沙箱安全；开发模式则必须允许 AI 对选中的项目目录拥有读写权限。因此需要单独设计项目白名单、权限确认、任务状态和 Git 保护机制。

## 核心判断

### 是否需要独立会话

需要，但这里的“独立会话”不应该理解为“复制一份项目到独立工作目录”。

建议拆成两层：

- **对话/任务上下文隔离**：每次开发任务仍然有独立的 chat session，用于保存消息、日志、权限记录、运行状态、附件和导出内容。
- **代码工作目录指向真实项目**：AI 的 `cwd` 和可写 `--add-dir` 指向用户从白名单中选择的 PC 项目路径，最终修改真实项目代码。

这样既保留了当前系统的多会话管理能力，又满足“手机控制 PC 项目开发”的真实目标。

### 是否需要独立工作树

第一版不强制创建独立工作树，直接在真实项目目录中工作，但必须加保护：

- 进入开发模式前显示当前项目路径和 Git 状态。
- 如果工作区已有未提交改动，明确提示用户。
- 每个任务结束时提供 diff、测试结果、提交/不提交选项。
- 删除聊天会话只删除对话记录，不删除项目代码。

后续增强版可以支持 `git worktree` 或临时分支模式：

- “直接修改当前分支”：最快，适合个人本地项目。
- “创建临时分支”：更安全，适合较大任务。
- “创建 git worktree”：隔离最好，但实现复杂度更高。

## 用户体验草图

### 入口

在当前聊天输入区或顶部增加一个“开发项目”按钮。

点击后打开项目选择面板：

- 项目名称
- 本机路径
- 当前分支
- 是否有未提交改动
- 最近使用时间

选择项目后，当前会话进入“开发模式”，页面顶部显示当前项目，例如：

`开发中：claude-web-server · main · 有未提交改动`

### 开发模式中的常用操作

- 发送开发需求：例如“帮我修复移动端输入框遮挡问题”。
- 查看 AI 思考和执行过程。
- 查看工具调用、命令输出、文件改动。
- 一键查看当前 diff。
- 一键运行预设测试。
- 停止当前任务。
- 允许或拒绝高风险操作。
- 任务完成后选择提交、继续修改或放弃由用户手动处理。

## 项目白名单设计

建议新增一个本地配置文件，默认不提交到 Git：

`claude_web_projects.config.json`

示例：

```json
{
  "projects": [
    {
      "id": "claude-web-server",
      "name": "Claude Web Server",
      "path": "D:/Code/claude-web-server",
      "default_tests": ["python -m compileall -q server.py claude_web"],
      "allow_git_commit": true,
      "allow_git_push": false
    }
  ]
}
```

服务端只允许选择白名单中的项目。前端永远不允许用户手输任意本机路径。

## 后端能力规划

### 新增 API

建议增加以下接口：

- `GET /api/dev/projects`：列出白名单项目和基础 Git 状态。
- `POST /api/dev/sessions/<session_id>/attach-project`：把当前聊天会话绑定到某个项目。
- `GET /api/dev/sessions/<session_id>/status`：查看当前开发任务状态。
- `GET /api/dev/sessions/<session_id>/diff`：返回当前项目 diff。
- `POST /api/dev/sessions/<session_id>/run-test`：运行项目预设测试。
- `POST /api/dev/sessions/<session_id>/stop`：停止当前 AI 开发任务。
- `POST /api/dev/sessions/<session_id>/permission/<request_id>`：批准或拒绝权限请求。

### 会话数据

在当前 `cache/<ip>/<user_id>/<session_id>/` 下保存开发任务元数据：

```json
{
  "mode": "development",
  "project_id": "claude-web-server",
  "project_path": "D:/Code/claude-web-server",
  "started_at": "...",
  "last_git_status": "...",
  "permissions": []
}
```

这里保存的是任务状态，不保存项目代码。

### CLI 启动策略

开发模式下，Claude CLI 的工作目录应切到项目根目录：

- `cwd = <project_path>`
- 可写目录：`<project_path>` 和当前会话 cache 目录
- 只读目录：全局只读能力目录、按需 skill/bundle 目录
- Prompt 明确说明：当前任务正在真实项目目录中开发，修改会直接落到 PC 本地项目

普通聊天模式保持现有逻辑，不受影响。

## 权限确认机制

这是这个功能最关键的部分。

### 权限分级

建议把操作分成三类：

#### 自动允许

低风险只读操作，可以直接执行：

- 读取项目内文件
- 搜索文件内容
- 查看 Git 状态
- 查看 diff

#### 每次确认

有副作用但常见的操作，需要手机端确认：

- 修改文件
- 安装依赖
- 运行测试命令
- 删除项目内文件
- 执行数据库迁移
- 创建 commit

#### 默认禁止

高风险操作默认禁止，除非后续专门增加白名单：

- 删除项目根目录
- 修改白名单之外的路径
- 执行 `git reset --hard`
- 执行 `git clean -fd`
- 推送到远程仓库
- 修改系统环境变量或全局配置

### 手机端交互

当 AI 请求敏感操作时，服务端通过 SSE 推送：

```json
{
  "type": "permission_request",
  "request_id": "...",
  "title": "允许运行测试命令？",
  "detail": "python -m compileall -q server.py claude_web",
  "risk": "medium",
  "options": ["allow_once", "deny"]
}
```

手机端显示底部弹窗：

- 操作类型
- 命令或文件路径
- 风险等级
- 允许一次
- 拒绝
- 停止任务

第一版不建议做“本会话永远允许”，避免手机上误点造成连续风险。等权限策略稳定后再加。

### 实现难点

如果 Claude CLI 本身能以结构化事件暴露权限请求，服务端可以直接转发给前端，并在用户选择后继续喂给 CLI。

如果 CLI 只在终端里弹交互式确认，则需要二选一：

- 使用 PTY/伪终端捕获交互提示，再把用户选择写回进程 stdin。
- 在开发模式中仍使用受控的 bypass 权限，但由服务端在工具调用层做自己的拦截和确认。

MVP 更现实的路线是先使用服务端自定义权限策略拦截高风险能力，再逐步适配 CLI 原生权限事件。

## 前端改造计划

### 桌面端

- 聊天页增加“开发项目”按钮。
- 增加项目选择弹窗。
- 消息流中增加命令输出、diff 摘要、权限请求卡片。
- 增加“查看 diff”“运行测试”“停止任务”快捷按钮。

### 手机端

手机端重点不是堆功能，而是让关键控制动作好点：

- 项目状态固定在顶部较小区域。
- 工具调用和日志默认折叠。
- diff 用文件列表优先展示，点击后再看具体 patch。
- 权限请求用底部弹窗，按钮足够大。
- 长命令输出默认只显示尾部，支持展开。

## Git 保护策略

进入开发模式时记录初始 Git 状态：

- 当前分支
- 是否有未提交改动
- untracked 文件数量
- 最近 commit

任务结束后展示：

- 修改文件列表
- diff 摘要
- 测试是否通过
- 是否需要提交

提交和推送分开处理：

- commit 可以作为第一阶段可选能力。
- push 第一阶段建议默认禁用，只在配置中显式允许。

## 推荐落地顺序

1. 新增项目白名单配置和 `/api/dev/projects`。
2. 前端增加“开发项目”入口和项目选择弹窗。
3. 支持把当前聊天会话绑定到项目，并让开发模式下的 CLI `cwd` 指向项目根目录。
4. 增加 Git 状态读取和 diff 查看。
5. 增加任务停止能力。
6. 增加预设测试命令执行。
7. 增加权限请求 UI 和服务端权限策略。
8. 最后再考虑 commit、push、临时分支、git worktree。

## 第一版验收标准

- 手机浏览器可以选择一个白名单项目。
- AI 能读取并修改该项目代码。
- 修改结果真实落到 PC 本地项目目录。
- 用户可以在手机上查看修改文件列表和 diff。
- 用户可以在手机上触发预设测试。
- 高风险命令不会静默执行。
- 停止任务后，CLI 子进程能被正确终止。
- 普通聊天模式不受开发模式影响。

## 暂不做的内容

- 不做原生移动 App。
- 不做任意路径输入。
- 不做多人同时编辑同一个项目的冲突解决。
- 不默认自动 push。
- 不默认为每个任务创建 git worktree。
- 不把项目源码复制到 cache 中。

## 主要风险

- Claude CLI 权限请求是否能稳定结构化转发到 Web。
- Windows 下长时间运行子进程、停止任务、PTY 交互的兼容性。
- 手机端查看大 diff 的体验。
- 用户本地项目已有脏改动时，AI 修改和人工修改混在一起。
- 允许真实项目写入后，权限边界必须比普通聊天更严格。

## 结论

这个功能可行，而且非常适合当前项目继续演进。建议第一版把目标控制在“手机选择白名单项目 -> 发开发任务 -> AI 修改真实项目 -> 手机查看 diff 和测试结果 -> 用户决定是否继续”。

独立的是任务上下文，不是项目工作目录。真实项目读写是这个功能的核心，但必须用白名单、Git 状态、diff、停止任务和权限确认把风险框住。
